# v0.3 Block G — Web UI (FastAPI + HTMX)

## Context

v0.2 shipped the full pair-and-triangle contradiction pipeline behind a Typer CLI. v0.3 Block G replaces the "edit YAML, run CLI, open markdown" loop with a single localhost web app: drop docs in, watch the run progress live, click into each contradiction.

Architecturally the web app is **a presentation layer over the existing pipeline** — it does not duplicate any business logic. `pipeline.ingest`, `pipeline.check`, the assertion store, and the audit logger stay canonical; the HTTP layer just routes requests to them.

## Decisions (confirmed)

1. **Stack**: FastAPI + Jinja2 server-rendered templates, with HTMX partials for navigation and live updates. No SPA, no React, no JS build step. Adds `fastapi`, `python-multipart`, `jinja2`, `httpx` (test client only).
2. **Information architecture**: contradictions tab is the index route (`GET /`). Other tabs are siblings, navigated via HTMX `hx-get` swaps into a `<main>` slot. Page reloads only happen on form `POST` followed by `303 See Other`.
3. **Diff button**: side-by-side assertion cards. Each card shows the assertion text, its parent document title + path, NLI `p_contradiction`, judge rationale, and quoted evidence spans. Multi-party diffs are three columns. No token-level highlighting in v0.3 (defer to v0.4 with a tiny diff library if needed).
4. **Live updates**: HTMX polling every 2 s against `/runs/{id}/stats.json`. Replaced by the final summary card when `run_status = "done"`. Server-Sent Events deferred to v0.4 if polling is too laggy under load.
5. **Run scope**: Contradictions tab always shows the latest completed run. No run-picker dropdown in v0.3 — keep the surface minimal; historical runs are still queryable via the CLI (`consistency-check report --run <id>`).
6. **Persistence**: same SQLite + FAISS as the CLI. Uploaded files land in `data_dir / "uploads" / <upload_id>` (timestamp + short hash). No automatic GC in v0.3 — flagged for v0.4.
7. **Auth**: none. Default bind `127.0.0.1:8000`. Threat model documented in ADR-0007; if anyone proposes binding `0.0.0.0` we add auth first.
8. **Background tasks**: FastAPI `BackgroundTasks` for `pipeline.check`. New `run_status` column on `pipeline_runs` (`pending | running | done | failed`). `ingest` stays synchronous in v0.3 — small uploads are already fast; deferred to v0.4 only if real corpora cause browser timeouts.

### Template naming convention

All Jinja2 templates and HTMX partials live under `consistency_checker/web/templates/` and are prefixed `cc_` so a mixed-app deployment can't collide with another framework's `contradictions.html` / `documents.html` / `stats.html`. Partials (HTMX fragments) keep the leading underscore *after* the prefix: `cc__pair_diff.html`. Routes stay clean — only filenames are namespaced.

### Output naming convention

Every report and export written by **either** the CLI or the web UI gets a unique, descriptive filename so two runs against the same corpus never overwrite each other. A shared helper `consistency_checker/audit/naming.py` produces:

- Reports — `cc_report_{run_started_at:%Y-%m-%dT%H-%M-%S}_{run_id_short8}.md`. Example: `cc_report_2026-05-14T10-30-00_a1b2c3d4.md`. Embeds when the run started (sortable) and 8 chars of the run id (uniquely identifies the run).
- CSV exports — `cc_assertions_{now:%Y-%m-%dT%H-%M-%S}.csv`. No run id (exports are snapshots, not per-run).
- JSONL exports — `cc_assertions_{now:%Y-%m-%dT%H-%M-%S}.jsonl`.
- Web-UI downloads inherit the same names via `Content-Disposition: attachment; filename="..."`.

CLI behaviour changes: `--out` becomes **optional**. When omitted, the file is written to `data_dir / "reports" / <generated_name>` (created on demand) and the resolved path is echoed on success. Explicit `--out` still wins. This change ships as **step G0b** (small standalone PR between G0 and G1) so the convention is in place before the web UI starts emitting downloads.

### G0b — Output naming helper + optional `--out` on the CLI
- New `consistency_checker/audit/naming.py` exposes `report_filename(run_id, started_at) -> str`, `export_csv_filename(now=...) -> str`, `export_jsonl_filename(now=...) -> str`. All return the `cc_*` formats above. Pure functions; no IO.
- `consistency-check report` and `consistency-check export {csv|jsonl}` change `--out` from required to optional. When omitted, the path is `data_dir / "reports" / <generated_name>`; the resolved path is echoed on success. Existing `--out=<path>` invocations behave identically.
- Files: `consistency_checker/audit/naming.py` (new), `consistency_checker/cli/main.py`, `tests/test_naming.py` (new), `tests/test_cli.py` (extended).
- Verify: hermetic table-driven tests for each helper; CLI test invokes `report` with no `--out`, asserts a file with the expected name pattern landed under `data_dir/reports/`.
- Depends: G0. Block G's later steps reuse the same helper.

### G1 — FastAPI skeleton + ingest tab
- `consistency_checker/web/app.py` exposes `create_app(config: Config) -> FastAPI`. Routes:
  - `GET /` — placeholder index, redirects to Contradictions tab (G2 replaces).
  - `GET /tabs/ingest` — HTMX partial: upload form + list of uploaded files for the current `data_dir`.
  - `POST /uploads` — multipart endpoint, writes files into `data_dir / "uploads" / <upload_id>`, then calls `pipeline.ingest` synchronously, then HTMX swaps in a success card.
- Base layout: `templates/cc_base.html` with HTMX-aware tab strip (Contradictions / Documents / Assertions / Stats / Ingest), each tab is `hx-get="/tabs/<name>" hx-target="#tab-content"`.
- Files: `consistency_checker/web/__init__.py`, `consistency_checker/web/app.py`, `consistency_checker/web/templates/{cc_base.html,cc_ingest.html,cc__upload_success.html}`, `consistency_checker/web/static/{cc_style.css,htmx.min.js}` (vendored — no CDN dep), `pyproject.toml` (deps), `tests/test_web_ingest.py`.
- Verify: hermetic `TestClient` test that posts a tiny `.txt` upload, asserts the file landed under `uploads/`, the ingest pipeline ran (assertions present in DB), and the success card is in the response body.

### G2 — Contradictions tab (main) + Diff button
- `GET /` → Contradictions tab. Pulls findings for `most_recent_run()` (uses `AuditLogger.most_recent_run` from D1).
- Two sections in the partial: "Pair contradictions" (rendered from `findings` where verdict ∈ `{contradiction, numeric_short_circuit}`) and "Multi-document conditional contradictions" (rendered from `multi_party_findings` where verdict = `multi_party_contradiction`).
- Each finding row has a Diff button: `hx-get="/findings/{id}/diff"` for pair, `hx-get="/multi_party_findings/{id}/diff"` for triangle, swapping into a `<dialog>` element.
- The diff partials are the side-by-side cards (two for pair, three for triangle), each showing assertion text, doc title, NLI p_contradiction (pair only), judge rationale, evidence spans.
- Files: `consistency_checker/web/app.py`, `consistency_checker/web/templates/{cc_contradictions.html,cc__pair_diff.html,cc__multi_party_diff.html}`, `tests/test_web_contradictions.py`.
- Verify: hermetic test seeds the audit DB with one pair finding and one multi-party finding, asserts both render on `GET /` and both Diff partials return the expected HTML structure.

### G3 — Documents + Assertions tabs
- `GET /tabs/documents` → paginated table of `documents` rows. Click → `GET /documents/{doc_id}` partial showing source path, metadata, and a chunk-level preview (re-using the loader).
- `GET /tabs/assertions` → paginated table of `assertions`. Click → partial showing the assertion text plus the surrounding chunk text and `char_start:char_end` slice.
- Files: `consistency_checker/web/app.py`, `consistency_checker/web/templates/{cc_documents.html,cc__document_detail.html,cc_assertions.html,cc__assertion_detail.html}`, `tests/test_web_browse.py`.
- Verify: hermetic test seeds 3 docs / 9 assertions, asserts paging works (`?page=2`), asserts the per-assertion view contains the surrounding chunk text.

### G4 — Stats tab (live + final)
- `GET /tabs/stats` → if a run is in progress (`run_status = "running"`), render a stats panel with HTMX `hx-trigger="every 2s" hx-get="/runs/{id}/stats.json"`. The polled endpoint returns a small HTML fragment with the current counters from `pipeline_runs` plus any new counter columns added in G5 (`n_assertions_processed`, `n_pairs_gated_so_far`, etc.).
- If the run is `done`, render the final summary card (matches `CheckResult` shape) and stop polling (HTMX stops via `hx-swap-oob` from the server emitting `HX-Trigger: stats-done`).
- Files: `consistency_checker/web/app.py`, `consistency_checker/web/templates/{cc_stats.html,cc__stats_live.html,cc__stats_final.html}`, `tests/test_web_stats.py`.
- Verify: hermetic test using a fixture pipeline that records intermediate counters; assert the live partial reflects mid-run state and the final partial reflects post-run state.

### G5 — Background tasks + `serve` command
- Migration `0004_run_status.sql` adds `run_status TEXT NOT NULL DEFAULT 'pending'` plus a handful of counter columns the live stats tab consumes.
- `pipeline.check` writes `run_status = "running"` at start and `"done"` / `"failed"` at end. New `AuditLogger.update_run_status(run_id, status)` helper.
- `POST /runs` (form with `--deep` checkbox + threshold sliders): creates the run row, returns `202 Accepted` with `HX-Redirect: /tabs/stats?run_id=...`, kicks off `pipeline.check` via FastAPI `BackgroundTasks`.
- CLI gets `consistency-check serve --host 127.0.0.1 --port 8000` that calls `uvicorn.run(create_app(cfg))`.
- Files: `consistency_checker/index/migrations/0004_run_status.sql`, `consistency_checker/audit/logger.py`, `consistency_checker/pipeline.py`, `consistency_checker/web/app.py`, `consistency_checker/cli/main.py`, `tests/test_web_background.py`, `tests/test_serve_cli.py`, `README.md`, `docs/ARCHITECTURE.md`, `CHANGELOG.md`.
- Verify: hermetic background-task test using `BackgroundTasks` in a real `TestClient` context (FastAPI guarantees they run synchronously when triggered through TestClient); assert `run_status` transitions through `running` → `done` and the final stats partial is reachable.

## Merge order

```
G0 → G0b → G1 → G2 → G3 → G4 → G5
```

G0b (the output-naming helper + CLI `--out` becoming optional) lands between G0 and G1 because both the CLI and the web UI rely on the same `naming.py` helper. G2 depends on G1 (base layout + HTMX wiring). G3 and G4 are independent of each other after G2 lands; either order works. G5 must come last because the run-status migration and `pipeline.check` instrumentation are load-bearing for the live stats UI.

## Critical files to be modified

- `consistency_checker/web/app.py` — every step touches it (additive route by route)
- `consistency_checker/web/templates/base.html` — base HTMX layout (G1, frozen after)
- `consistency_checker/pipeline.py` — `check` gains `run_status` writes in G5
- `consistency_checker/index/migrations/0004_run_status.sql` — additive schema (G5)
- `consistency_checker/cli/main.py` — `serve` subcommand (G5)

## End-to-end verification (post-G5)

```sh
# Start the server
uv run consistency-check serve --host 127.0.0.1 --port 8000 &
# Browser: http://127.0.0.1:8000
# 1. Click Ingest tab, drop a .pdf + a .docx + 2 .md files, submit.
# 2. Click Run, leave --deep checked, submit.
# 3. Watch the Stats tab tick up: assertions extracted, pairs gated, findings.
# 4. When run_status flips to "done", Contradictions tab auto-populates.
# 5. Click Diff on the canonical vacation-policy triangle → 3-column drill-in.
# 6. Documents tab shows the 4 uploaded files; Assertions tab shows N atomic facts.
```

Pass-bar: the canonical 4-file mixed-format upload completes end-to-end through the UI, the Stats tab live-updates at ≤ 2 s latency, and the planted triangle contradiction appears in the Contradictions tab with the Diff button rendering the three contributing assertions correctly.

## Out of scope (deferred to v0.4)

- Token-level diff highlighting inside the Diff button.
- Server-Sent Events for live stats (HTMX polling first; revisit if 2 s latency feels bad).
- Run picker (Contradictions tab is latest-run-only in v0.3).
- Authentication. Bind `127.0.0.1` only and document the threat model.
- Upload garbage collection. `data/uploads/` grows unbounded in v0.3.
- Async ingest. v0.3 keeps ingest synchronous because real upload sizes are small; revisit if real corpora cause browser timeouts.
- Multi-user / multi-tenant support. v0.3 is single-user, single-data_dir.
